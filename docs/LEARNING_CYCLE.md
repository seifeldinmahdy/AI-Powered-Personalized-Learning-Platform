# Learning Cycle — How the Platform Actually Works

This document traces the **real, implemented** flow from the moment a student enrolls
through sessions, labs, and problem sets — what gets generated, **where it is stored**,
how it is generated, and what each step reads and writes. Everything here is taken from
the code (file:line references included), not from assumptions.

> Scope note (as requested): this covers the learning cycle (pathway → session → lab →
> problem set → profile/mastery). It deliberately does **not** cover the talking-avatar
> (TTS/A2F) or the intent classifier internals.

---

## 0. The three runtimes and where data lives

| Runtime | Role | Persistence it owns |
|---|---|---|
| **Django** (`backend/`, `:8000`) | System of record for users, courses, enrollment, lesson completion, gamification, and the **learning profile** | PostgreSQL/SQLite via Django ORM |
| **AI service** (`ai_service/`, FastAPI, `:8001`) | All generation + analysis (slides, labs, problem sets, profilers, pathway router host) | **JSON files** + **one SQLite db** under `ai_service/data/` and `course_pathway/` |
| **Frontend** (`frontend/`, Vite/React) | Orchestrates the flow; calls Django **and** the AI service directly | `localStorage` (auth) + `sessionStorage` (pathway plan & slide cache) |

**Everything generated by AI is stored as files by the AI service** (not in Django),
except the **learning profile** and **concept mastery**, which the AI service pushes into
Django. Concretely:

| Artifact | Stored at | Format |
|---|---|---|
| Placement-derived student context | `ai_service/data/student_contexts/{student}_{course}.json` | JSON (`UnifiedStudentContext`) |
| Pathway / session plan (authoritative) | `course_pathway` SQLite (`session_plans` table) | JSON blob per (student, course) |
| Pathway copy for the UI gate | Django `Enrollment.current_pathway` (JSONField) | JSON |
| Pathway copy the live session reads | Browser `sessionStorage['pathway_plan']` | JSON |
| Generated slides | Browser `sessionStorage['slides_cache_{lessonId}']` + transient `SharedSessionStore` | JSON (**not saved to Django**) |
| Live session state (emotions, tutor events, slide content) | `SharedSessionStore` (in-memory, AI service) | Pydantic objects |
| Coding lab | `ai_service/data/coding_labs/{student}_{course}_{lesson}.json` | JSON (`CodingLabGenerateResponse`) |
| Problem set + submissions + hints | `ai_service/data/problem_sets/{student}/{lesson}/{ps_id}.json` | JSON (`ProblemSetData`) |
| Profiler audit trail | `ai_service/data/profile_audit/{student}/audit.json` | JSON (append) |
| Learning profile (`profile_summary`, `profile_data`, `concept_mastery`) | Django `StudentLearningProfile` | ORM row |
| Lesson completion + XP/streak | Django `LessonCompletion`, `StudentProfile`, `DailyStudyStats` | ORM rows |

### Service-to-service auth
When the AI service reads/writes Django it cannot use the student's JWT. It sends two
headers and Django resolves the user from them:

```python
# backend/apps/core/authentication.py:40
def authenticate(self, request):
    service_key = request.META.get("HTTP_X_SERVICE_KEY", "")
    student_id  = request.META.get("HTTP_X_STUDENT_ID", "")
    if not service_key:           return None          # fall through to JWT
    if service_key != INTERNAL_SERVICE_KEY: raise AuthenticationFailed("Invalid service key")
    user = User.objects.get(pk=int(student_id))        # "impersonate" that student
    return (user, "internal-service")
```

So `X-Service-Key` + `X-Student-ID` = "act as this student". The frontend, by contrast,
calls Django with the student's `Authorization: Bearer <jwt>`.

## The cycle at a glance

```
enroll ─▶ placement test ─▶ UnifiedStudentContext (JSON)
   │
   ▼
generate pathway ─▶ SessionPlan (sqlite) + Enrollment.current_pathway + sessionStorage
   │
   ▼  for each lesson:
   ├─ LIVE SESSION  ─ fetch session chunks ─▶ generate slides ─▶ SharedSessionStore
   │     └─ on end: mark LessonCompletion(Completed) ─▶ gamification signal (XP, progress)
   │     └─ fire-and-forget SESSION PROFILER ─▶ Django StudentLearningProfile
   ├─ CODING LAB    ─ generate notebook lab ─▶ coding_labs JSON
   │     └─ on complete: LAB PROFILER ─▶ Django StudentLearningProfile (PATCH)
   └─ PROBLEM SET   ─ generate questions ─▶ problem_sets JSON
         └─ per submit: binary rubric eval ─▶ score (Python) ─▶ concept_mastery (PATCH)
         └─ on summary: PROBLEM-SET PROFILER ─▶ Django StudentLearningProfile (PATCH)
   │
   ▼ (last lesson complete → progress 100%) ─▶ capstone is the terminal gate
```

## 1. Enrollment

The student enrolls from the course page. `EnrollmentViewSet.perform_create` stamps the
student and points `current_lesson` at the first lesson:

```python
# backend/apps/courses/views.py:169
def perform_create(self, serializer):
    course = serializer.validated_data.get("course")
    first_lesson = Lesson.objects.filter(module__course=course)\
        .order_by("module__module_order", "lesson_order").first()
    serializer.save(student=self.request.user, current_lesson=first_lesson)
```

After enrolling, the UI routes the student into the **assessment** (placement) unless the
pathway is already marked ready. The lesson route is gated by `RequirePathway`, which
redirects to the assessment when `is_pathway_ready` is false:

```tsx

    return <Navigate to={`/courses/${courseId}/assessment`} replace />;
}
```

## 2. Placement test → `UnifiedStudentContext`

### 2.1 Generating the quiz
`Assessment.tsx` marks the enrollment as started, then asks the AI service for questions
(`POST /assessments/generate-categorized`), which runs:
ChromaDB topics → semantic dedup → LLM grouping (5 categories) → per-category questions
(`ai_service/routers/assessments.py:94`).

```tsx
// frontend/src/pages/Assessment.tsx:139
await api.patch(`/courses/enrollments/${enrollmentId}/`, { is_assessment_started: true });
```

### 2.2 Submitting the quiz — the single source of student context
On submit, the frontend posts every answer to the AI service **and** writes the score back
to the Django enrollment:

```tsx
// frontend/src/pages/Assessment.tsx:171
const result = await submitPlacementResults({
    student_id: String(studentId), course_id: String(id), course_title: courseTitle,
    enrollment_id: enrollmentId ?? 0, composition_mode, language_proficiency,
    answers: answerPayload,
});
if (enrollmentId) { await updatePlacementScore(enrollmentId, result.score_pct); }
```

`POST /assessments/submit-placement` is the one endpoint that converts raw answers into the
context every downstream component reads. It computes the score, mastery tier, per-topic
performance, strengths/weaknesses, the list of wrong answers, builds a
`UnifiedStudentContext`, and **persists it to a JSON file**:

```python
# backend/.. no — ai_service/routers/assessments.py:128
score_pct = round((correct_count / total) * 100)
mastery_level = "Expert" if score_pct >= 70 else "Intermediate" if score_pct >= 40 else "Novice"
# per-topic correctness → topic_performance; strengths >0.7, weaknesses <0.5
profile = StudentProfileState(student_id=..., course_id=..., mastery_level=mastery_level,
    strengths=strengths, weaknesses=weaknesses, topic_performance=topic_performance,
    incorrectly_answered=incorrectly_answered, course_intent=req.course_title, ...)
context = UnifiedStudentContext(profile=profile, live=LiveSessionState())
get_student_context_store().save(req.student_id, req.course_id, context)   # → JSON file
```

Storage is a flat JSON file keyed by student+course:

```python
# ai_service/services/student_context_store.py:35
def _key_path(self, student_id, course_id):
    safe_key = f"{student_id}_{course_id}".replace("/", "_").replace("\\", "_")
    return self._dir / f"{safe_key}.json"   # ai_service/data/student_contexts/
```

`UnifiedStudentContext` has two halves (`ai_service/schemas/student_context.py`):
- `profile` (`StudentProfileState`) — slow-changing: mastery_level, composition_mode,
  language_proficiency, strengths, weaknesses, topic_performance, incorrectly_answered,
  `student_profile_summary`.
- `live` (`LiveSessionState`) — fast-changing real-time fields used during a session
  (current slide, running summary, tutor_events, emotion_signals, time_spent_per_slide…).

> Note: the `course_id` written here is the **Django course id as a string** (the frontend
> sends `course_id: String(id)`). The same string is later used to look up content in
> ChromaDB (see §3.2).

### 2.3 Topic performance is updated again later
Two endpoints re-apply a weighted moving average to `topic_performance` after in-session
MCQ checkpoints (`/student-context/{id}/{course}/update-performance` and
`/assessments/submit`). Both recompute strengths/weaknesses and rewrite
`student_profile_summary`, but **never** touch mastery_level / composition_mode /
language_proficiency (`ai_service/routers/student_context.py:97`,
`ai_service/routers/assessments.py:326`).

## 3. Pathway generation

### 3.1 The frontend trigger
`CoursePathway.tsx` runs on mount: it reads the persisted context from the AI service,
calls `POST /pathway/generate`, then **syncs the plan back to Django** and finally caches
the plan in `sessionStorage` when the student clicks "Begin Course".

```tsx
// frontend/src/pages/student/CoursePathway.tsx:80
const ctxRes = await fetch(`${AI}/student-context/${studentId}/${courseId}`);
const contextProfile = (await ctxRes.json()).profile;

const result = await generatePathway({
    student_id: contextProfile?.student_id || String(studentId),
    course_id:  contextProfile?.course_id  || 'pythonlearn',   // ← context course_id (Django id) or fallback
    mastery_level: contextProfile?.mastery_level || 'Novice',
    strengths: contextProfile?.strengths || [], weaknesses: contextProfile?.weaknesses || [],
    topic_performance: contextProfile?.topic_performance || {}, ...
    use_synthetic_context: false,
});

// push to Django so RequirePathway will pass
await api.post(`/courses/enrollments/${enrollment.id}/save_pathway/`, { pathway: result });
// ...later, on "Begin Course":
sessionStorage.setItem('pathway_plan', JSON.stringify(plan));
navigate(`/course/${courseId}/lesson/${firstLessonId}`);
```

### 3.2 The pipeline (AI service)
`POST /pathway/generate` builds a `StudentContext` and calls `PathwayGenerator.generate()`.
It is **cached by a hash of the context** — same inputs return the stored plan instantly:

```python
# course_pathway/src/pathway/generator.py:115
ctx_hash = context.context_hash()
if not force_regenerate and not self._store.needs_regeneration(student_id, course_id, ctx_hash):
    cached = self._store.load(student_id, course_id)
    if cached is not None: return PathwayResponse(plan=cached, cached=True)

chunks = self._reader.get_all_course_chunks(context.course_id)   # ← from ChromaDB
if not chunks: raise ValueError(f"No chunks found for course '{context.course_id}'")

# LLM designs the curriculum from the topic tags, then:
sections, section_chunks_raw = self._build_sections_from_curriculum(curriculum, chunks_by_topic)
personalised_sections, section_chunks = self._personalizer.personalize(sections, chunks, context)
sessions = self._grouper.group_sessions(personalised_sections, section_chunks)  # token-budgeted
plan = SessionPlan(student_id=..., course_id=..., sessions=sessions, ...)
self._store.save(plan)                                            # ← SQLite
return PathwayResponse(plan=plan, cached=False)
```

Pipeline stages (all under `course_pathway/src/pathway/`):
1. **ChromaDB read** — `chromadb_reader.get_all_course_chunks(course_id)` returns
   `CourseChunk`s (raw_text, topic, page range, difficulty, is_definitional).
2. **LLM curriculum design** — `llm/curriculum.design_curriculum(...)` orders topics into
   sessions (falls back to alphabetical if no LLM key).
3. **Section building** — chunks grouped per session topic, deduped, definitional-first.
4. **Personalizer** — selects chunks per section based on the student context.
5. **SessionGrouper** — packs sections into sessions by a token budget.
6. **Persist** — `PlanStore.save(plan)`.

### 3.3 Where the plan is stored (three copies)
**(a) Authoritative — SQLite**, one row per (student, course), upserted with the context hash:

```python
# course_pathway/src/pathway/storage/plan_store.py:57
INSERT INTO session_plans (student_id, course_id, plan_json, context_hash, created_at)
VALUES (?, ?, ?, ?, ?)
ON CONFLICT (student_id, course_id) DO UPDATE SET plan_json=excluded.plan_json, ...
```

**(b) Django** — `save_pathway` stores the plan JSON on the enrollment and flips the gate.
Note it **receives a `slides` field but deliberately discards it** (no slide persistence):

```python
# backend/apps/courses/views.py:176
@action(detail=True, methods=["post"], url_path="save_pathway")
def save_pathway(self, request, pk=None):
    enrollment = self.get_object()
    pathway_data = request.data.get("pathway", {})
    slides_data  = request.data.get("slides", [])
    with transaction.atomic():
        enrollment.current_pathway = pathway_data
        enrollment.is_pathway_ready = True
        enrollment.save(update_fields=["current_pathway", "is_pathway_ready"])
        # "Here we would normally save the slides that are generated." → pass (NOT saved)
    return Response({"status": "pathway and slides saved", "is_pathway_ready": True})
```

**(c) Browser** — `sessionStorage['pathway_plan']`, which is the copy the live session reads.

## 4. The live session

Route `course/:courseId/lesson/:lessonId` (gated by `RequirePathway`) renders
`LiveSession.tsx`. Each session gets a fresh UUID `session_id` (`sessionIdRef`).

### 4.1 Loading slides for the session
The session reads the cached plan, finds the matching session, and either uses cached
slides or generates them from the session's chunks:

```tsx
// frontend/src/pages/student/LiveSession.tsx:167
const pathwayPlan = JSON.parse(sessionStorage.getItem('pathway_plan'));
const currentSession = pathwayPlan.sessions.find(s => s.session_number === sessionNum);

const cachedSlides = sessionStorage.getItem(`slides_cache_${lessonId}`);
if (!cachedSlides) {
    // 1) fetch the raw chunks for this session from the cached plan + ChromaDB metadata
    const chunks = await (await fetch(`${AI}/pathway/session-chunks`, { method:'POST',
        body: JSON.stringify({ student_id, course_id, session_number: sessionNum }) })).json();
    // 2) ask the slides generator to turn chunks into slides
    const slideResponse = await generateSlides({ session_number, session_title,
        topics_covered, book, chunks, mastery_level:'Novice',
        composition_mode:'visual_heavy', language_proficiency:'Elementary' });
    setSlides(slideResponse.slides);
    sessionStorage.setItem(`slides_cache_${lessonId}`, JSON.stringify(slideResponse.slides)); // cache
}
```

`/pathway/session-chunks` loads the cached plan, finds the session, and returns each
chunk's `raw_text` plus topic/page metadata looked up in ChromaDB
(`course_pathway/router.py:233`).

> **Slides are never persisted server-side.** They live only in `sessionStorage` (per
> lesson) and, transiently, in the in-memory `SharedSessionStore` (current slide only).
> The Django `Slide` model is **not** written by this flow.

### 4.2 Live session state — `SharedSessionStore`
As the student moves through slides, the frontend PATCHes the AI service session state.
This is an **in-memory singleton** (optionally Redis); it holds the running emotion/tutor
log that the session profiler later analyzes.

```tsx
// frontend/src/pages/student/LiveSession.tsx:243 (and on every slide nav)
await fetch(`${AI}/session/${sessionIdRef.current}`, { method:'PATCH', body: JSON.stringify({
    current_slide_index: 0, current_slide_title: slides[0].title,
    current_slide_content: slides[0].body_content?.map(i=>i.text).join('\n'),
    current_topic: currentSession.session_title, visited_slides_push: 0 })});
```

```python
# ai_service/services/session_store.py:84  (singleton, threading.Lock)
def create_session(self, session_id, profile, live=None, ttl_seconds=86400): ...
def update_session(self, session_id, live_kwargs=None, profile_kwargs=None): ...
```

The PATCH endpoint **auto-creates** the session if the frontend writes before any
`/tutor/start` (`ai_service/routers/session.py:88`). Emotion fusion (`/profiler/fuse-emotions`)
also appends to `live.emotion_signals` here.

### 4.3 Ending the session → lesson completion
When the student finishes the slides, `LiveSession` marks the lesson **Completed** in
Django and routes to the lab:

```tsx
// frontend/src/pages/student/LiveSession.tsx:588
const existing = completions.find(c => String(c.lesson) === String(lesson.id));
if (existing) result = await markLessonComplete(existing.id, undefined, timeSpentMinutes);
else {
    const created = await createLessonCompletion({ enrollment: enrollment.id,
        lesson: lesson.id, status: 'Completed' });
    result = await markLessonComplete(created.id, undefined, timeSpentMinutes);
}
// ...then route into the lesson-end coding lab
navigate(`/course/${courseId}/lesson/${lesson.id}/lab`, { state: { nextLessonId: nextLesson?.id ?? null, ... }});
```

`markLessonComplete` calls `POST /progress/lesson-completions/{id}/complete/`, which fires
the gamification signal (XP, streak, and — when all lessons are done — progress 100%).
See §8.

## 5. The session profiler (fire-and-forget)

Right before completion, `LiveSession` runs the **session profiler**. This is the only
profiler whose result is written to Django **by the frontend** (the lab and problem-set
profilers write to Django themselves). Sequence:

```tsx
// frontend/src/pages/student/LiveSession.tsx:439
// 1. read the current profile from Django (with the student's JWT)
const existing = await (await fetch(`${API}/progress/learning-profile/`, {headers:{Authorization:`Bearer ${token}`}})).json();
// 2. ask the AI service to rewrite it from the live session log
const profilerData = await (await fetch(`${AI}/profiler/update`, { method:'POST', body: JSON.stringify({
    student_id, lesson_title, session_id: sessionIdRef.current,
    existing_profile_summary: existing.profile_summary, existing_profile_data: existing.profile_data })})).json();
// 3. overwrite the profile back in Django (increments sessions_count)
await fetch(`${API}/progress/learning-profile/`, { method:'POST', headers:{Authorization:`Bearer ${token}`},
    body: JSON.stringify({ sessions_count: existingSessionsCount + 1,
        profile_summary: profilerData.profile_summary,
        profile_data: { profile_summary: profilerData.profile_summary, ...profilerData.profile_data }})});
```

On the AI side, `/profiler/update` pulls the live log straight from `SharedSessionStore`
(by `session_id`), groups it **per slide**, and runs an LLM "profile rewriter":

```python
# ai_service/routers/profiler.py:174
if request.session_id:
    session_data = store.get_session(request.session_id)
    log_dicts = _group_session_log_by_slide(session_data)   # per-slide: emotions, questions, tutor turns, time
result = await update_profile(student_id, lesson_title, session_log=log_dicts,
    existing_profile_summary=..., existing_profile_data=..., session_id=...)
```

`update_profile` (`ai_service/services/profiler_service.py:297`) sends the existing profile
+ per-slide log to the LLM with `PROFILE_REWRITER_SYSTEM`, which returns a full **rewrite**
of `profile_summary` + `profile_data` (learning_style_signals, topics_of_difficulty/strength,
unresolved_questions, recurrent_mistakes, …) plus an audit array `_changes_made` that is
stripped before returning and written to the audit log
(`ai_service/data/profile_audit/{student}/audit.json`).

## 6. The coding lab

After the session, `CodingLab.tsx` calls `POST /api/coding/labs/generate`.

### 6.1 Generation + caching
The lab is keyed by student+course+lesson and cached on disk; it is only regenerated on
`force_regenerate`:

```python
# ai_service/services/lab_service.py:490
async def generate_coding_lab(request):
    lab_id = store.lab_id(request.student_id, request.course_id, request.lesson_id)
    if not request.force_regenerate:
        cached = store.load(lab_id)
        if cached: return cached (cached=True)
    # ... two-step LLM build: (1) checklist from slides+profile, (2) notebook cells
    # plus per-cell "suggested questions" (_generate_suggested_questions)
    response = CodingLabGenerateResponse(lab_id=lab_id, cached=False, checklist=..., lab=...)
    store.save(response)     # → ai_service/data/coding_labs/{lab_id}.json
    return response
```

The lab is a notebook of `explanation` / `code` / `task` cells. The generator pulls the
student's profile from Django first (`_fetch_student_profile`, via `X-Student-ID`).

### 6.2 During the lab
The student can run code (`POST /api/coding/labs/run` — sandboxed subprocess), get spoken
narration, write notes, and reveal "suggested questions". Each of these is written into the
**same lab JSON file** (`ai_service/services/lab_store.py`):

```python
# ai_service/services/lab_store.py
save_cell_note(lab_id, cell_id, content, timestamp)   # appends to cell.student_notes
save_general_note(lab_id, content, timestamp)         # appends to lab.general_notes
mark_question_asked(lab_id, cell_id, question_text)   # sets was_asked=True
```

### 6.3 Completing the lab → lab profiler
`POST /api/coding/labs/complete` stamps `completed_at` and fires the **lab profiler** as a
background task:

```python
# ai_service/routers/coding.py:169
store.mark_completed(req.lab_id)
asyncio.create_task(run_lab_profiler(student_id=..., lab_id=..., course_id=..., lesson_id=...))
```

`run_lab_profiler` (`ai_service/services/profiler_service.py:576`) reads the lab notes,
which questions were asked vs ignored, and task completion; **fetches** the current profile
from Django; runs the LLM with `LAB_PROFILER_SYSTEM` (which **appends** to list fields and
rewrites `profile_summary`); writes the audit entry; and **PATCHes Django itself**:

```python
# ai_service/services/profiler_service.py:554
async def _patch_student_profile(student_id, profile_data):
    await client.patch(f"{DJANGO_API_URL}/progress/learning-profile/update/",
        json={"profile_data": profile_data},
        headers={"X-Student-ID": student_id, "X-Service-Key": service_key})
```

The frontend then routes from the lab to the problem set.

## 7. The problem set

`ProblemSet.tsx` calls `POST /problem-set/generate` on mount.

### 7.1 Generation — grounded in the session
The problem set is generated from the **actual session content** (slides + lab cells),
the student's profile, and the course's concepts. The generator fetches concepts +
weak-concepts and asks the LLM to tag each rubric criterion with a `concept_id`:

```python
# ai_service/services/problem_set_service.py:283 (generate)
profile_data = await _fetch_student_profile(student_id)          # Django
course_concepts = await fetch_course_concepts(course_id)         # Django /courses/{id}/concepts
concept_mastery = await fetch_concept_mastery(student_id)        # Django learning-profile
weak = top_weak_concepts(concept_mastery, n=3)                   # lowest scores first
# prompt includes slides_summary, lab_summary, profile_context, weak_concept_block, concepts_block
# → LLM returns questions, each with a rubric of criteria→checks (binary yes/no) + concept_id
problem_set = ProblemSetData(problem_set_id=uuid, student_id, lesson_id, course_id, questions=...)
store.save(problem_set)   # → ai_service/data/problem_sets/{student}/{lesson}/{ps_id}.json
```

Hallucinated concept ids are dropped so only real course concepts survive
(`problem_set_service.py`, the `valid_concept_ids` filter).

### 7.2 Submitting an answer — score is computed in Python
`POST /problem-set/submit` runs the **binary-rubric evaluator**: the LLM only fills
`result: true/false` + `evidence` per check; the numeric score is computed in code, with
hint penalties already applied per check:

```python
# ai_service/services/problem_set_service.py:546 (_calculate_score)
for criterion in evaluated_criteria:
    for check in criterion["checks"]:
        contribution = criterion["weight"] * check["weight"]
        if check["result"] is True:
            total += max(contribution - hint_deductions.get(check["id"], 0.0), 0.0)
# final_score = round(total); passed = final_score >= 65
```

The submission (code, hints_used, evaluated rubric, score) is saved into the problem-set
JSON. Then the router **fires the concept-mastery update as a background task**:

```python
# ai_service/routers/problem_set.py:71
asyncio.create_task(update_concept_mastery_from_eval(
    student_id=request.student_id,
    evaluated_rubric=[c.model_dump() for c in result.evaluated_rubric]))
```

### 7.3 The mastery loop (the only writer of `concept_mastery`)
`mastery.py` converts binary check results into EMA-smoothed per-concept scores and PATCHes
them into Django's `concept_mastery` field — never the LLM:

```python
# ai_service/services/mastery.py
def update(old, outcome, alpha=0.3): return round(old + alpha*(outcome - old), 4)  # EMA
# compute_mastery_updates(): per criterion → average its checks' 1.0/0.0 → build entry per concept_id
async def patch_concept_mastery(student_id, updates):
    await client.patch(f"{DJANGO_API_URL}/progress/learning-profile/update/",
        json={"concept_mastery": updates}, headers={"X-Student-ID":..., "X-Service-Key":...})
```

### 7.4 Hints
`POST /problem-set/hint` (levels 2 and 3) finds the highest-impact failing check, generates
a targeted hint, and **records an escalating penalty** into the problem-set JSON
(`save_hint_deduction`, `problem_set_store.py:88`). Those deductions are what
`_calculate_score` subtracts in §7.2.

### 7.5 Reaching the summary → problem-set profiler
When the student lands on the summary screen, the frontend calls
`POST /problem-set/summary-viewed`, which fires the **problem-set profiler** in the
background:

```python

```

`run_problem_set_profiler` (`profiler_service.py:704`) aggregates rubric-category failures,
hint-3 usage, score trajectory, and recurrent mistake tags across the whole set, fetches
the current profile, runs the LLM with `PROBLEM_SET_PROFILER_SYSTEM` (append-only enrichment),
writes the audit entry, and PATCHes Django `profile_data`.

The summary's "next" button goes to the next session (or, on the last lesson, the capstone
entry — see the capstone docs). The lesson was already marked complete at the end of the
**live session** (§4.3), not here.

## 8. Progression & course completion

`POST /progress/lesson-completions/{id}/complete/` sets the row to Completed and saves,
which triggers the gamification signal:

```python

@receiver(post_save, sender=LessonCompletion)

    if instance.status != "Completed" or not instance.completed_at: return
    profile = user.student_profile                         # requires StudentProfile to exist
    # +XP (50, +100 first lesson), streak, daily stats...
    total_lessons = enrollment.course.total_lessons_count
    completed_in_course = LessonCompletion.objects.filter(enrollment=enrollment, status="Completed").count()
    if completed_in_course >= total_lessons:
        Enrollment.objects.filter(pk=enrollment.pk).update(progress_percentage=100.0)   # ← material done
        profile.current_xp += XP_COURSE_COMPLETE
```

So **material completion = all lessons Completed → `progress_percentage = 100`**. For a
course with a capstone, the course is only *truly* complete when the capstone is PASSED —
that gate lives in `apps/courses/completion.py` and the capstone app (documented separately).

## 9. The learning profile & mastery — who writes what

There is **one** `StudentLearningProfile` row per student with three independently-owned
pieces:

| Field | Written by | How |
|---|---|---|
| `profile_summary` + `profile_data` (qualitative) | **Session profiler** (frontend POSTs overwrite, `sessions_count += 1`) | `POST /progress/learning-profile/` |
| `profile_data` (enriched, append-only) | **Lab profiler** + **Problem-set profiler** (AI service) | `PATCH /progress/learning-profile/update/` with `profile_data` |
| `concept_mastery` (numeric per-concept EMA) | **mastery.py only** (AI service, after each problem-set submit) | `PATCH /progress/learning-profile/update/` with `concept_mastery` |

The PATCH handler merges each key independently, so the three writers never clobber each
other (`backend/apps/progress/views.py:165` `partial_update`). The invariant that the
profiler LLM can never overwrite numeric mastery is enforced because the profilers only ever
send `profile_data`, never `concept_mastery` (`profiler_service.py:554`).

`concept_mastery` is then read back by: the dashboard chart
(`GET /progress/concept-mastery/`), the problem-set generator (weak-concept targeting),
CLO attainment, and the capstone team role advisor.

## 12. Factual notes / gotchas observed in the code (not assumptions)

1. **Slides are never persisted server-side.** `save_pathway` accepts a `slides` payload
   and discards it (`backend/apps/courses/views.py:188` is a `pass`); slides live only in
   `sessionStorage` (`slides_cache_{lessonId}`) and transiently in `SharedSessionStore`.
2. **The lesson is marked complete at the end of the live session**, before the lab and
   problem set (`LiveSession.tsx:588`). The lab/problem set still run, but `progress_percentage`
   already advanced.
3. **`SharedSessionStore` is in-memory** (per AI-service process). Restarting the AI service
   loses live session state; the session profiler depends on it being present when
   `/profiler/update` is called.
4. **The pathway is cached by context hash.** Re-running `/pathway/generate` with the same
   student context returns the stored plan; it only regenerates when the context changes or
   `/pathway/regenerate` is used (`generator.py:115`, `plan_store.py:113`).
5. **`course_id` is the Django course id string throughout the AI side** (set at placement,
   `Assessment.tsx:173`), and that same string is used to look up ChromaDB content; if no
   chunks exist for it, `/pathway/generate` raises `No chunks found` (`generator.py:161`).
6. **Three profilers, one profile, different write paths.** Session profiler = frontend POST
   overwrite; lab + problem-set profilers = AI-service PATCH (append). All three append to a
   single audit log per student.
7. **`concept_mastery` is written by `mastery.py`** after a problem-set submit (the lesson
   loop). The **capstone adds two more deterministic writers** in Django — see §13.7 and
   §13.11. The qualitative profilers still only ever send `profile_data`.

# 13. The Capstone — the terminal stage

The capstone is **not** part of the per-lesson loop. It is a separate, course-level stage
that begins **after the coursework is finished** and acts as the **terminal completion
gate**: a course that has a capstone is only "complete" when the capstone is **PASSED**
(§13.9). Everything in §1–§8 feeds it — the capstone reads the same `concept_mastery` the
problem-set loop built, and writes back to it.

All capstone state lives in **Django** (`apps/capstone/`); the student's code lives in a
**GitHub repo**; the platform stores only `repo_url + commit_sha + results JSON` — **never
the code**.

## 13.0 Where capstone data lives

| Artifact | Stored at | Notes |
|---|---|---|
| Capstone definition + rubric | Django `Capstone`, `CapstoneRubricItem` | admin-authored; rubric items carry `category` (core/stretch), `concept` FK, `clo` FK, `weight`, `min_team_size` |
| Submission (repo, sha, results, score, verdict) | Django `CapstoneSubmission` | **no code stored** — only `repo_url`, `latest_commit_sha`, `results` JSON, `score`, `verdict` |
| Student code | GitHub repo (`<org>/capstone-<id>-<username>`) | public repo; `work` branch + branch-protected `main` |
| Team + role advice | Django `Team` (`role_advice` JSONField) | advisory only |
| Matchmaking queue | Django `MatchmakingQueueEntry` | `fill_window_expires_at` so the queue never deadlocks |
| AI-assist quota + log | Django `CapstoneAssistQuota`, `CapstoneAssistLog` | per-student credits |
| Idempotency bookkeeping | `CapstoneSubmission.xp_awarded`, `mastery_applied` | rewards granted once on first PASS |

## 13.1 Admin prerequisite (so there is something to trigger into)
A capstone only exists if an admin created one for the course and set it `status="active"`.
The rubric can be AI-drafted (`POST /capstone/capstones/{id}/draft-rubric` →
`capstone_rubric_service.draft_core_criteria`) but the **LLM never emits a score or verdict**
— it only proposes binary yes/no criteria. Admins review/edit them. Each criterion is tagged
`core` or `stretch`, and optionally linked to a `concept` and a `clo`.

A student can reach the capstone page (`/course/:id/capstone`) for any course where they're
enrolled **and** a capstone is `active`/`completed`:

```python
# backend/apps/capstone/views.py:519
def capstone_for_course(request, course_id):
    enrolled = Enrollment.objects.filter(student=user, course_id=course_id).exists()
    cap = Capstone.objects.filter(course_id=course_id, status__in=["active","completed"])\
        .prefetch_related("rubric_items").first()
```

> The capstone **page** is not hard-gated on progress — but the **entry point / trigger**
> that surfaces it (next section) is.

## 13.2 THE TRIGGER — when the student enters the capstone

The capstone is surfaced by a single React component, `CapstoneStartCTA`, which renders
**only when the course material is at 100%** and the course actually has a capstone. The
gate is the same `progress_percentage == 100` that the gamification signal sets when the
last lesson completes (§8).

It is wired into **three places**:


## 13.4 In-platform IDE — read / commit / CI verdict / run

The workspace (`CapstoneWorkspace.tsx`) talks only to Django, which performs Git operations
via the **GitHub Git Data API** (`capstone_git.py`):

| Action | Endpoint | What happens server-side |
|---|---|---|
| Load file tree | `GET /capstone/capstones/{id}/tree/` | `get_tree(repo, work)` |
| Open a file | `GET /capstone/capstones/{id}/file/?path=` | `get_file(repo, work, path)` |
| Commit | `POST /capstone/capstones/{id}/commit/` | atomic blobs→tree→commit→move-ref on **`work`** (never `main`), retried on stale ref; rate-limited 20/hr |

| Run (local) | `POST /capstone/capstones/{id}/run/` | proxied to the AI sandbox; **feedback only, not the verdict** |

Committing to `work` triggers GitHub Actions (CI) on that branch; the workspace polls the
verdict endpoint. This is **continuous feedback**, not grading.

## 13.5 THE GRADING TRIGGER — "Submit for grading"

This is the action that turns a repo into a graded, scored submission. The student clicks
**Submit for grading** in the IDE (after committing CI-green work). It runs:

```python
# backend/apps/capstone/views.py:876
def submit_for_grading(request, capstone_id):
    sub, _cap, err = _resolve_submission(request.user, capstone_id)        # must have a repo
    work_sha = head_sha(sub.repo_url, sub.branch)                          # HEAD of 'work'
    verdict = get_check_runs(sub.repo_url, work_sha)                       # CI must be green
    if verdict["status"] != "completed" or verdict["conclusion"] != "success":
        return Response({"error": "CI must pass …", "verdict": verdict}, status=409)
    move_ref(sub.repo_url, "main", work_sha, force=False)                  # fast-forward main → that commit
    sub.latest_commit_sha = work_sha; sub.status = "evaluating"; sub.save(...)
    threading.Thread(target=_grade_from_repo, args=(sub.id, work_sha), daemon=True).start()
    return Response({"status": "evaluating", "commit_sha": work_sha, ...}, status=202)
```

So the trigger chain is: **CI green on `work` HEAD → fast-forward `main` to that exact
commit → grade `main` in a background thread.** Branch protection stays satisfied because
the SHA already passed the required `ci` check. The graded code is read transiently and
never stored:

```python
# backend/apps/capstone/views.py:835
def _grade_from_repo(submission_id, sha):
    bundle = read_repo_bundle(sub.repo_url, sha)     # text files → in-memory bundle (capped), NOT persisted
    _evaluate_and_grade(sub, bundle, proposal_text)
```

## 13.6 Deterministic grading + verdict (the scoring invariant)

`_evaluate_and_grade` calls the AI evaluator, which returns **binary pass/fail + evidence
per rubric item only**; the **score and PASS/FAIL verdict are computed in Python**:

```python
# backend/apps/capstone/views.py:213
rubric_items = _get_effective_rubric(capstone, team_size)   # size-scaled: solo→core-only-applicable
ai_result   = _call_ai_evaluate(capstone, rubric_items, code_bundle, proposal_text)  # LLM: {id:{passed,evidence}}
results = _normalize_results(rubric_items, ai_result)
score   = _compute_score(rubric_items, results)            # weighted % in Python
verdict = _compute_verdict(rubric_items, results, capstone.pass_policy)
```

```python
# backend/apps/capstone/views.py:75  (verdict is Python, never the LLM)
def _compute_verdict(rubric_items, results, pass_policy="all_core"):
    core = [i for i in rubric_items if i.category == "core"]
    if not core: return "pass"
    return "pass" if all(results.get(str(i.id),{}).get("passed") for i in core) else "fail"
```

**PASS = every applicable `core` criterion passed.** Stretch criteria move the score/feedback
but never the verdict. This is uniform across solo/team and admin-defined/student-proposed.

## 13.7 Rewards — granted exactly once, on first PASS

`_apply_grade` always overwrites the stored result (so re-submitting re-scores and the
status never locks), but the **rewards bundle fires only on the first PASS** and never
multiplies:

```python
# backend/apps/capstone/views.py:171
grant_rewards = verdict == "pass" and not sub.mastery_applied
if grant_rewards:
    sub.mastery_applied = True
    sub.xp_awarded = int(CAPSTONE_XP * score / 100)
sub.save(...)
if grant_rewards:
    _apply_xp_delta(student_id, sub.xp_awarded)                 # capstone XP → StudentProfile
    _update_concept_mastery_sync(student_id, results, rubric_items)  # ← 2nd writer of concept_mastery
    mark_complete_if_eligible(sub.enrollment)                   # ← terminal completion gate
```

This is where the capstone **feeds back into the §9 profile/mastery system**:
- `_update_concept_mastery_sync` (`views.py:110`) applies the same EMA used by the lesson
  loop, but **in Django, in-process**, keyed by each rubric item's `concept` FK. So
  `concept_mastery` has a **second deterministic writer** (the capstone), refining §9.
- `mark_complete_if_eligible` stamps `Enrollment.completed_at` (§13.9).

## 13.8 The other submission paths (what they actually do)

There are three ways a `CapstoneSubmission` can be graded/recorded — important to be precise:

1. **`submit_for_grading`** (§13.5) — the IDE path. **This is the authoritative graded path.**
2. **`submit_archive`** (`views.py:552`) — the "paste your code" path on the capstone page.
   It runs the **same** `_evaluate_and_grade` synchronously on the pasted bundle and returns
   the graded submission. Also deterministic, also rewards-once.
3. **`submit_from_repo`** (`views.py:789`) — the "paste repo URL + commit SHA" path. It only
   **records** `repo_url + commit_sha` and sets `status="evaluating"`; it does **not** run the
   rubric evaluator. Its completion is driven solely by the webhook below.

**The webhook does not grade.** `github_webhook` (`views.py:745`) verifies the HMAC
signature and, on a `check_suite` conclusion, only flips a matching submission's *status*:

```python
# backend/apps/capstone/views.py:745
if event == "check_suite" and conclusion in ("success", "failure"):
    CapstoneSubmission.objects.filter(repo_url=repo_url, latest_commit_sha=head_sha, status="evaluating")\
        .update(status="completed" if conclusion=="success" else "failed", evaluated_at=now)
```

So CI status (webhook) and the rubric verdict (`submit_for_grading` / `submit_archive`) are
separate signals; the rubric score/`verdict` only comes from paths 1 and 2.

## 13.10 Teams + matchmaking + role advice (when role advice triggers)

For `team_mode="team"` capstones:
- Students **join the queue** (`queue_join`); `process_queue` (matchmaking) forms teams over
  `concept_mastery` complementarity, or a **solo team on fill-window timeout** (never
  deadlocks).
- **Role advice triggers at team formation.** When `process_queue` creates a team with **≥2
  members**, it fires generation in the background:

```python

    from .team_roles import trigger_async
    trigger_async(team)        # → POST /capstone/team-roles on the AI service, caches result on Team.role_advice
```

`team_roles.generate_for_team` builds each member's **course-restricted `concept_mastery`**
and the rubric, and asks the AI service for an **advisory** lead/support division of labor.
It degrades gracefully to a balanced rotation when evidence is thin, and **never feeds
scoring** (text only). It is also regenerated on demand via
`POST /capstone/team/{id}/role-advice/refresh/`.

   active; the `progress==100` gate applies to the *CTA that surfaces it*, not the page.
2. **Two graded paths, one webhook-status path.** Only `submit_for_grading` and
   `submit_archive` produce a rubric `score`/`verdict`; `submit_from_repo` + the webhook only
   move `status`.
3. **The verdict and score are always computed in Python** (`_compute_verdict`,
   `_compute_score`); the LLM returns binary pass/fail + evidence only.
4. **Rewards are idempotent** via `mastery_applied`/`xp_awarded`: re-submitting re-scores but
   never re-awards XP/mastery or re-stamps completion.
5. **`concept_mastery` now has three writers total**: the AI `mastery.py` (problem sets,
   §7.3), the capstone grade on first PASS (`_update_concept_mastery_sync`, Django), and the
   capstone assist penalty (`_apply_assist_mastery_penalty`, Django). All write the same
   field via independent PATCH/ORM paths — refines §9.
6. **No student code is ever stored** — only `repo_url`, `commit_sha`, and the `results` JSON.
   The grading bundle is read transiently in `read_repo_bundle` and discarded.
7. **`move_ref` to `main` requires the GitHub App to be able to update the protected branch**
   (Administration permission); it relies on the SHA already having a green `ci` check. This
   is the one capstone step that cannot be verified without live GitHub credentials.

# 14. The real-time interaction layer (Tutor · Intent · Emotion · RAG)

This is what happens **inside a live session** while the student is on a slide — the part
that produces the talking tutor ("Dr. Nova"), reacts to questions, reads the student's
emotion, and grounds answers in the textbook. It runs entirely on the AI service and the
frontend; **nothing here writes to Django directly.** Its sole durable output is the
material it accumulates in `SharedSessionStore`, which the **session profiler** (§5) reads
at the end of the session to rewrite the Django profile.

It is orchestrated by one React widget — `CompactTutor.tsx` — overlaid on the slides in
`LiveSession`, sharing the session's UUID (`sessionId={sessionIdRef.current}`).

## 14.0 Two "sessions" share one id

| "Session" | Where | Holds | Keyed by |
|---|---|---|---|
| **`TutorSession`** | `tutor_service.py` (in-memory dataclass) | lecture state machine: topics→subtopics, `running_summary`, `transcript`, `pace_modifier` | the same `session_id` |
| **`SharedSessionStore`** entry | `session_store.py` (in-memory `UnifiedStudentContext`) | live slide content, `emotion_signals`, `tutor_events`, `running_summary`, current topic | the same `session_id` |

The tutor writes its state **into** the shared store after every step, so intent/emotion/
profiler can read up-to-date context without the frontend passing it:

```python
# ai_service/services/tutor_service.py:518
def _sync_to_shared_store(session):
    store.update_session(session.session_id,
        live_kwargs={"current_topic": ..., "current_subtopic": ...,
                     "running_summary": session.running_summary,
                     "tutor_transcript": session.transcript[-10:],
                     "pace_modifier": session.pace_modifier},
        profile_kwargs={"student_profile_summary": session.student_profile_summary or ""})
```

## 14.1 The tutor (Dr. Nova) — a lecture state machine

`POST /tutor/start` creates a `TutorSession` from a list of topics (auto-generating four
subtopics per topic if none are supplied) and, if a `student_id` is given, **fetches the
Django learning profile** + weak concepts and attaches them for personalization
(`tutor_service.py:573`, `create_session:598`).

The lecture advances chunk-by-chunk:

```
POST /tutor/continue  (loop)            POST /tutor/ask  (mid-lecture question)
  generate_lecture_chunk()                answer_question()
    → LLM writes the next subtopic          → LLM answers using running_summary + history
    → _update_summary() compresses          → does NOT advance the topic pointer
       running_summary (recursive)          → Socratic skills applied (see 14.2)
    → _advance() moves the pointer
    → TTS audio (+ A2F blendshapes)         both return text (+ audio) and sync to the store
```

The **running summary** is a recursively-compressed memory so the model keeps context
without replaying the whole transcript:

```python
# ai_service/services/tutor_service.py:1505
async def _update_summary(session, new_content):
    prompt = f"EXISTING SUMMARY:\n{session.running_summary or '(none yet)'}\n\nNEW CONTENT…"
    session.running_summary = await _call_ollama(SUMMARIZE_SYSTEM_PROMPT, prompt, temperature=0.2, num_predict=400)
```

`/tutor/continue` and `/tutor/ask` also synthesize TTS, and the **student's current fused
emotion is mapped to TTS prosody** (rate/pitch) so delivery mirrors tone
(`tutor.py:412` `_EMOTION_TTS_MAP`, e.g. `confused → -15% rate`). `/tutor/set-pace` and the
`/tutor/repeat` (verbatim/rephrase) endpoints adjust delivery without advancing the pointer.

## 14.2 Tutor "skills" — composable system-prompt fragments

The tutor's behaviour is steered by named skill fragments in `TUTOR_SKILLS`
(`tutor_service.py:140`). The final system prompt = base prompt + the active skills:

```python
# ai_service/services/tutor_service.py:284
def _build_system_prompt(base_prompt, active_skills):
    parts = [base_prompt.strip()]
    for k in active_skills:
        if k in TUTOR_SKILLS: parts.append(TUTOR_SKILLS[k])
    return "\n\n".join(parts)
```

Which skills activate is decided per call from the **profile, emotion, and intent**:
- **First chunk** → `BACKGROUND_PROBE`. A profile present → `ENGAGEMENT_ADAPT`.
- **Profile-driven** (lecture, `tutor_service.py:790`): `DIFFICULTY_TOPIC` / `STRENGTH_TOPIC`
  (from `topics_of_difficulty`/`strength`), `VISUAL_LEARNER` / `HANDS_ON_LEARNER` (from
  `learning_style_signals`), `PACE_SLOW`/`PACE_FAST`, `SURFACE_UNRESOLVED` (weaves in an
  open question from a prior session that matches the current subtopic), `RECURRENT_MISTAKE`.
  A specific profile skill overrides the generic `ENGAGEMENT_ADAPT`.
- **Q&A** (`answer_question`, `tutor_service.py:1070`): `SOCRATIC_GUARD` always (never give
  the direct answer), escalating to `SOCRATIC_SCAFFOLD` after the student fails the same
  question twice.
- **Intent-driven**: `EMOTIONAL_ACKNOWLEDGEMENT` (Emotional-State), `PACE_ACKNOWLEDGEMENT`
  (Pace-Related). These are the same `SOCRATIC_GUARD`/`SOCRATIC_SCAFFOLD` fragments the
  capstone assistant reuses (§ capstone assist).

## 14.3 Intent classifier — routes every student utterance

A TinyBert-CNN classifies the student's text into **five** pedagogical intents; a confidence
gate demotes low-confidence predictions to `Unknown`:

```python
# ai_service/services/intent_service.py:117
intent_names = ['On-Topic Question','Off-Topic Question','Emotional-State','Pace-Related','Repeat/clarification']
if max_confidence < confidence_threshold:   # default 0.65
    results.append({"intent_name": "Unknown", "raw_prediction": predicted_name, ...})
```

`POST /intent/classify` auto-fills the model's **context string** from `SharedSessionStore`
when the caller doesn't pass one (`intent.py:33` → `session_store.build_context_string`,
which emits `topic:… | prev:… | ability:… | emotion:… | pace:… | slides:…`,
`session_store.py:251`). Off-topic detection lives here now — the old `/tutor/relevance`
is deprecated and always returns `True` (`tutor_service.py:504`).

## 14.4 Emotion — FER + SER → fusion → session store

Two recognizers feed a fusion step:
- **FER (facial):** the webcam sends a frame every ~25 s to `POST /fer/predict` (single
  image) or `/fer/predict-video` (rolling buffer that averages frames for stability)
  (`fer.py:18`). Labels: Angry/Disgust/Fear/Happy/Neutral/Sad/Surprise.
- **SER (speech):** audio is posted to `POST /ser/predict`.

The frontend **fuses** them (`emotionFusion.ts:31`): agree → use it; only one present → use
it; both missing → `neutral`; **conflict → ask the AI service to arbitrate** (LLM, 3 s
timeout, confidence fallback):

```tsx
// frontend/src/pages/student/LiveSession.tsx:382 (poll loop, ~25s)
const fusion = await fuseEmotions(
  { fer_emotion, fer_confidence, ser_emotion, ser_confidence },
  { slide_index: currentSlide, subtopic: lesson?.title, session_id: sessionIdRef.current });
setFusedEmotion(fusion.fused_emotion);
```

`POST /profiler/fuse-emotions` runs the LLM arbiter (`profiler_service.fuse_emotions`,
`FUSION_SYSTEM`) **and writes the result back into `SharedSessionStore`** — setting
`fused_emotion` and appending a structured entry to `live.emotion_signals`
(`profiler.py:259`). That `emotion_signals` log is exactly what the **session profiler**
groups per slide at session end (§5). The current fused emotion is also handed to the tutor
as `student_emotion` (drives TTS prosody + emotion skills).

> The fusion LLM is told that the FER/SER label `"uncertain"` means the recognizer could not
> classify — it is treated as **missing data**, never as student confusion
> (`profiler_service.py:127`).

## 14.5 RAG question answering — grounded answers with citations

`POST /rag/ask` runs a retrieval-augmented engine over its **own** ChromaDB
(`rag_pipeline/data/chroma`), returning an answer plus source citations (book, page range,
topic, relevance), with an in-process LRU cache (`rag.py:87`):

```python
# ai_service/routers/rag.py:99
response = engine.ask(question=..., course=..., topic=..., difficulty=..., top_k=5)
# → {answer, sources:[{book,page_start,page_end,topic,relevance_score}], question}
```

## 14.6 The in-session request flow (what actually happens on a question)

`CompactTutor.handleAsk` is the orchestrator (`CompactTutor.tsx:330`):

```
student speaks/types  q
  ├─ keyword override (repeat / pace / emotion phrases) — fast path, skips the model
  └─ else: intent = classifyIntent(q, "topic:<lesson> | prev:<lesson> | emotion:neutral | pace:normal")

route on intent:
  Off-Topic        → canned redirect + TTS (does not call the LLM tutor)
  Emotional-State  → askTutor("offer encouragement + re-explain simpler", emotion)
  Pace-Related     → setTutorPace(slow|fast|normal) + canned confirmation
  Repeat/clarif.   → replay the cached audio of the last chunk (else fall through to re-ask)
  On-Topic (default):
        ragRes = askRag(q, lessonTitle)                       # ← RAG FIRST
        if ragRes has real sources:
            augmentedQuestion = q + "\n\nRelevant textbook context:\n" + ragRes.answer
        res = askTutor(sessionId, augmentedQuestion, emotion) # Socratic tutor, grounded
```

So for a normal content question the path is **intent → RAG retrieval → tutor LLM answer
(Socratic, grounded in the retrieved textbook context, in the student's current emotion)**.
Confused-emotion can also **auto-trigger** an explanation (`isAutoTrigger`), with a staleness
guard that discards the answer if the student moved to a different slide while waiting
(`CompactTutor.tsx:505`).

## 14.7 How the realtime layer connects back to the cycle

Everything above is **transient** (in-memory on the AI service). Its one persistent
consequence is that `SharedSessionStore.live` accumulates `emotion_signals`, `tutor_events`,
`time_spent_per_slide`, and the `running_summary` during the session. When the student ends
the session, the **session profiler** (§5) reads that store by `session_id`, groups it per
slide, and rewrites the Django learning profile (`topics_of_difficulty`,
`unresolved_questions`, `recommended_approaches`, …). That profile then feeds the **next**
pathway/slide/lab/problem-set generation and the tutor's own skill selection — closing the
loop.

## 14.8 Triggers & transience

| Piece | Trigger | Endpoint / code | Persistence |
|---|---|---|---|
| Tutor lecture chunk | frontend loop while lecturing | `POST /tutor/continue` → `generate_lecture_chunk` | `TutorSession` (memory) + synced to store |
| Tutor answer | student asks a question | `POST /tutor/ask` → `answer_question` | memory + store |
| Intent classification | each student utterance | `POST /intent/classify` | none (returned to caller) |
| FER | webcam frame (~25 s) | `POST /fer/predict[-video]` | rolling buffer (memory) |
| SER | captured audio | `POST /ser/predict` | none |
| Emotion fusion | after FER/SER | `emotionFusion.ts` → `POST /profiler/fuse-emotions` | writes `fused_emotion` + `emotion_signals` to `SharedSessionStore` |
| RAG answer | On-Topic question | `POST /rag/ask` | in-process LRU cache only |
| Profile read for tutor | `/tutor/start` with `student_id` | `_fetch_student_profile_for_tutor` (Django GET) | read-only |

## 14.9 Factual notes / gotchas (realtime layer)

1. **This layer never writes Django.** It only mutates in-memory stores; the Django profile
   is written later by the session profiler (§5), which reads `SharedSessionStore`.
2. **RAG uses a separate ChromaDB** (`rag_pipeline/data/chroma`) from the pathway/slides
   ChromaDB (`course_pathway`). They are configured independently (`rag.py:55`).
3. **Two LLM "sessions" share one `session_id`** — the `TutorSession` (lecture machine) and
   the `SharedSessionStore` entry; the tutor syncs into the latter after every step.
4. **Off-topic is an intent class now**, not a separate relevance call; `/tutor/relevance`
   is a deprecated stub that always returns `True`.
5. **Low-confidence intents become `Unknown`** (threshold 0.65) so the tutor can ask the
   student to rephrase instead of acting on a wrong guess.
6. **`"uncertain"` emotion = missing data**, by explicit instruction to the fusion + profiler
   LLMs — it is never treated as confusion.
7. **The fused emotion has two live effects**: TTS prosody (rate/pitch) and tutor skill
   selection; it becomes durable signal only via the session profiler.
8. **The intent classifier and FER/SER are ML models** loaded as singletons in the AI
   service; if a model file/credential is missing the endpoints fail (intent → 503), and the
   frontend degrades (e.g. RAG/intent failures fall back to a plain tutor answer).
